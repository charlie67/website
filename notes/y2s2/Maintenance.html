Maintenance

The modification of software after delivery to correct faults, improve performance or change other attributes.

There are four types of maintenance:

  • Corrective: Fixing bugs in the program
  • Perfective: Adding or improving functionality
  • Adaptive: Modifying software to better meet the clients evolved needs
  • Preventative: Refactoring, defensive rewrites, protective changes

Influencing factors

Maintenance is the biggest and most expensive part of any software project, taking 50-70 percent of the total effort of much commercial software.

Cost is increased by:

  • Shitty programmers
  • Old code and technical debt
  • Changes to project
  • Poor organisation or structure

... And decreased by:

  • A simple design
  • Good documentation (Design documents, maintenance guide)
  • Automation where possible for consistency (e.g. JavaDoc, JUnit etc.)
  • Regression testing
  • Refactoring

Handling maintenance

Bug reports or change requests are usually needed for maintenance to be necessary. The process goes like this:

  • Get the report
  • Decide on the severity
  • If/when necessary, assign effort
  • Fix the problem and add tests for it
  • Update documentation accordingly

Refactoring

Refactoring is essentially fixing code before a problem has a chance to manifest itself. It's where you don't change what the underlying code does, but instead change the internal structure to make it easier to understand and easier to work with.

Refactoring can involve...

  • Renaming methods
  • Consolidating or splitting classes (Subclasses, superclasses or siblings)
  • Encapsulation

Although they are small steps, time spent refactoring a program when signs of bad code start to appear can save a lot of time and headache in the long run.

Signs of bad code, or a bad codebase

  • Duplicated code
  • Very large class(es)
  • Long methods (10+ lines usually, doing several things)
  • Switch statements (in OOP, often imply bad used of OO principles)
  • Complicated or long comments defining a function
